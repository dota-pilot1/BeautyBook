# AI 개발 방법론 — 정답 자판기 문제와 해결 전략

> 작성일: 2026-04-23  
> 목적: AI 보조 개발에서 발생하는 구조적 문제를 인식하고, 시너지를 최대화하는 개발 방식 정립

---

## AI는 정답 자판기다

AI는 인풋에 따라 빠르게 결과를 내준다.  
문제는 그 빠름 자체가 함정이 될 수 있다는 것이다.

---

## 핵심 문제 3가지

### 1. 인풋 의존성 — 내 인풋의 퀄리티가 곧 결과의 퀄리티

AI 아웃풋은 내 인풋을 그대로 증폭한다.  
잘못된 방향으로 인풋을 주면 **완성도 높은 오답**이 나온다.

인풋 퀄리티에 영향을 주는 요소:

| 영역 | 부족할 때 생기는 문제 |
|------|----------------------|
| 문법 / 언어 이해 | 엉뚱한 구현 방향 지시 |
| 아키텍처 이해 | 레이어 오염, 의존성 역전 |
| 비즈니스 로직 이해 | 도메인 규칙이 빠진 구현 |
| DB 설계 이해 | 정규화 오류, 인덱스 누락 |
| 디자인 감각 | UX 흐름 파악 없는 UI 요청 |
| 계획 수립 능력 | 순서 없는 개발 → 반복 수정 |
| 판단력 / 이해력 | AI 아웃풋 검증 불가 |
| 프롬프트 / 컨텍스트 엔지니어링 | 같은 요청도 결과 품질 천차만별 |

---

### 2. 블랙박스 구현 — 내부 과정 모르고 만들기

속전속결 구현은 내부 동작 이해 없이 결과만 붙여넣게 된다.

결과:
- 장애 발생 시 원인 파악 불가
- 요구사항 변경 시 수정 범위 예측 불가
- 확장 시 구조 붕괴

> "돌아가는 코드"와 "이해하는 코드"는 다르다.

---

### 3. 속도 중심 개발 — 컨벤션과 아키텍처 소홀

빠름에 집중하면 다음이 무너진다:

- 레이어 분리 원칙 위반
- 네이밍 컨벤션 불일치
- 중복 코드 난립
- 테스트 부재

초기 속도는 높지만, **유지보수 비용이 기하급수적으로 증가**한다.

---

## 해결 전략 — 검증된 프로토타입 + 보일러플레이트 기반 개발

### 핵심 원칙

> AI는 **생성** 도구가 아니라 **확장** 도구로 사용한다.

반복되는 구조(인증, 권한, 메뉴, 에러 처리 등)는  
미리 직접 이해하고 검증한 형태로 템플릿화해 둔다.  
AI는 그 위에서 도메인 로직만 확장하는 역할로 제한한다.

---

### 실천 방법

#### 보일러플레이트 / 프로토타입 사전 제작

직접 이해하고 검증한 구조를 도메인별로 미리 만들어 둔다:

| 영역 | 예시 |
|------|------|
| 인증 / 보안 | JWT Rotate, Spring Security, httpOnly Cookie |
| 권한 관리 | User → Role → Permission N:M RBAC |
| 공통 에러 처리 | ErrorCode enum + GlobalExceptionHandler |
| 메뉴 / 네비게이션 | DB 기반 트리 메뉴 + 권한 게이팅 |
| 배포 | Docker Compose, Nginx, GitHub Actions CI |
| 프론트엔드 구조 | FSD (Feature-Sliced Design) + TanStack Query |

#### AI 역할 제한

```
❌ AI에게 설계부터 구현까지 전부 맡기기
✅ 검증된 구조를 기반으로 AI에게 확장 요청하기

❌ "회원 관리 시스템 만들어줘"
✅ "이 User 엔티티 구조 기반으로 프로필 수정 API 추가해줘"
```

#### 아웃풋 검증 능력 유지

AI 아웃풋을 채택하기 전 스스로 판단할 수 있어야 한다:

- 이 구조가 우리 아키텍처 레이어 원칙에 맞는가?
- 비즈니스 규칙이 올바른 레이어에 있는가?
- 이 코드를 내가 설명할 수 있는가?

> 인풋 퀄리티를 높이는 것만큼, **아웃풋 검증 능력**도 중요하다.  
> AI가 생성한 결과를 보고 "이게 맞는가"를 판단하는 능력이 없으면,  
> 좋은 인풋을 넣어도 틀린 결과를 채택하게 된다.

---

## 이 전략이 강력한 이유

| 효과 | 설명 |
|------|------|
| 인풋 퀄리티 향상 | 도메인 설계 선행으로 명확한 요청 가능 |
| 아웃풋 검증 용이 | 기존 검증된 구조와 비교해 이상 탐지 가능 |
| 오류 범위 축소 | AI 역할을 "확장"으로 제한해 영향 범위 예측 가능 |
| 팀 일관성 유지 | 보일러플레이트 기반이므로 컨벤션 자동 유지 |
| 유지보수 비용 감소 | 구조가 안정되어 있어 수정 비용 예측 가능 |
