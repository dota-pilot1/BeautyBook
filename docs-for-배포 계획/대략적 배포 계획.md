# 대략적 배포 계획

## 1. 목적

이 문서는 `pilot-callcenter-container` 프로젝트의 초기 운영 배포 방향을 대략적으로 정리한 문서다.  
목표는 복잡한 분산 구조를 바로 도입하는 것이 아니라, **빠르게 배포 가능하고 관리 가능한 1차 운영 구조**를 잡는 것이다.

---

## 2. 현재 프로젝트 구성

현재 구조는 아래 3개 축으로 이해하면 된다.

- 프론트엔드: Next.js (`pilot-callcenter-front`)
- 백엔드: Spring Boot (`pilot-callcenter-server`)
- 데이터베이스: PostgreSQL (`docker-compose.yml`)

현재 로컬 개발 포트는 아래와 같다.

- Frontend: `4100`
- Backend: `4101`
- PostgreSQL: `5434`

---

## 3. 1차 배포 방향

가장 현실적인 1차 배포 방향은 아래와 같다.

- 배포 서버: AWS EC2 1대
- DB: PostgreSQL 컨테이너 운영
- 백엔드: Spring Boot JAR 실행
- 프론트엔드: Next.js production build 후 Node 서버 실행
- 리버스 프록시: Nginx

즉, **EC2 한 대에서 프론트 / 백엔드 / DB를 함께 운영하는 단일 서버 구조**를 1차 목표로 본다.

이 방식의 장점:

- 초기 구축이 단순함
- 비용 통제가 쉬움
- 로컬 구조와 유사해서 배포 난이도가 낮음
- 장애 포인트가 비교적 명확함

주의점:

- 서버 1대 장애 시 전체 서비스 영향
- DB까지 같은 서버에 있어 확장성은 낮음
- 트래픽 증가 시 프론트 / 백엔드 / DB 분리가 필요할 수 있음

---

## 4. 권장 배포 구조

### 외부 접근 흐름

```text
사용자 브라우저
  -> Nginx (80 / 443)
    -> Next.js (4100)
    -> Spring Boot API (4101)
    -> PostgreSQL (내부 접근만 허용)
```

### 구성 원칙

- PostgreSQL은 외부에 직접 노출하지 않음
- 외부 공개는 Nginx만 담당
- 프론트 요청은 Nginx를 통해 Next.js로 전달
- API 요청은 Nginx에서 Spring Boot로 프록시
- HTTPS는 Nginx + Let's Encrypt 또는 AWS 인증서 전략으로 처리

---

## 5. 배포 단위

### 프론트엔드

- `npm install`
- `npm run build`
- `npm run start`

운영 시:

- `pm2` 또는 `systemd` 로 프로세스 관리
- 기본 내부 포트는 `4100` 유지 가능

### 백엔드

- `./gradlew build`
- 생성된 JAR 실행

운영 시:

- `java -jar` 실행
- `pm2`, `systemd`, 또는 별도 서비스 파일로 관리
- 내부 포트는 `4101`

### DB

- 현재 `docker-compose.yml` 기준 PostgreSQL 컨테이너 사용
- 운영 초기에는 이 구성을 그대로 활용 가능

운영 시:

- Docker volume 으로 데이터 영속화
- DB 백업 정책 필요

---

## 6. 인프라 초안

### 서버

- AWS EC2 Ubuntu 24.04 LTS 또는 22.04 LTS
- 초반 인스턴스 예시:
  - `t3.small` 또는 `t3.medium`

### 스토리지

- EBS 사용
- 애플리케이션 코드, 로그, Docker volume, PostgreSQL 데이터 저장

### 네트워크

- 80 / 443 포트만 외부 공개
- 22 포트는 운영자 IP만 허용 권장
- 4100 / 4101 / 5432(또는 5434) 는 외부 직접 오픈하지 않음

---

## 7. 환경 변수 / 설정 항목

정확한 값은 추후 `.env` 또는 운영 설정 문서로 분리하고, 현재는 아래 정도를 배포 필수 설정 항목으로 본다.

### 프론트엔드

- API base URL
- 운영 도메인 URL

예시:

```env
NEXT_PUBLIC_API_BASE_URL=https://api.example.com
```

또는 같은 도메인에서 프록시할 경우:

```env
NEXT_PUBLIC_API_BASE_URL=/api
```

### 백엔드

- DB 접속 정보
- JWT secret
- CORS 허용 origin
- 운영 프로필

예시:

```env
SPRING_PROFILES_ACTIVE=prod
DB_HOST=127.0.0.1
DB_PORT=5432
DB_NAME=pilot_callcenter
DB_USERNAME=postgres
DB_PASSWORD=...
JWT_SECRET=...
ALLOWED_ORIGIN=https://서비스도메인
```

### DB

- DB 사용자 / 비밀번호
- 데이터 volume 경로
- 백업 보관 정책

## 8. 1차 배포 절차 초안

### 1단계. 서버 준비

- EC2 생성
- 보안 그룹 설정
- Docker, Docker Compose, Node.js, Java 설치
- Nginx 설치

### 2단계. DB 기동

- PostgreSQL 컨테이너 실행
- volume 생성 확인
- 접속 확인

### 3단계. 백엔드 배포

- 소스 배포 또는 빌드 산출물 업로드
- 운영 환경 변수 설정
- Spring Boot 실행
- 헬스 체크 및 Swagger 접근 확인

### 4단계. 프론트엔드 배포

- `npm install`
- `npm run build`
- `npm run start`
- 브라우저 접근 확인

### 5단계. 리버스 프록시 연결

- Nginx 설정
- `/api` -> Spring Boot
- 나머지 웹 요청 -> Next.js
- 도메인 연결 및 HTTPS 적용

### 6단계. 운영 안정화

- 재기동 정책 설정
- 로그 위치 정리
- 장애 시 재시작 절차 문서화

## 10. 추후 확장 방향

초기에는 EC2 1대 구조로 시작하되, 아래 시점에 분리를 고려한다.

### DB 분리

아래 상황이면 RDS PostgreSQL 전환 검토:

- 데이터 중요도가 올라감
- 백업 / 복구 안정성이 중요해짐
- 운영자 수동 관리 부담이 커짐

### 애플리케이션 분리

아래 상황이면 프론트 / 백엔드 분리 검토:

- 트래픽 증가
- 배포 주기가 서로 달라짐
- 서비스 장애 영향을 분리하고 싶음

### 컨테이너 표준화

아래 상황이면 프론트 / 백엔드 모두 Docker 기반 배포 검토:

- 배포 재현성을 높이고 싶음
- 서버 환경 차이를 줄이고 싶음
- CI/CD 자동화를 붙이고 싶음

---

## 11. 현 시점 결론

현재 프로젝트는 아래 방식이 가장 무난하다.

- **1차 배포 목표: EC2 1대 배포**
- **DB: PostgreSQL 컨테이너 유지**
- **백엔드: Spring Boot 단독 실행**
- **프론트엔드: Next.js production 실행**
- **Nginx: 외부 진입점 담당**

즉, 지금 단계에서는 **빠르게 올리고 운영 감각을 익히는 배포**가 우선이고,  
트래픽 / 안정성 요구가 커질 때 DB 분리와 CI/CD 고도화를 진행하는 방향이 적절하다.
