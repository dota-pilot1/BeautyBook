# 시그니처(Signature)와 팩토리 메서드(Factory Method)

---

## 시그니처란?

메서드나 생성자의 **이름 + 파라미터 목록**을 합쳐서 시그니처라고 한다.
"어떤 파라미터를 받는지 정의"를 뜻하는 말이다.

```java
// fromClaims()의 시그니처
public static UserPrincipal fromClaims(
    Long id, String email, String username,
    String roleCode, List<String> permissions  ← 파라미터 목록
)
```

시그니처를 변경한다 = 파라미터를 추가/제거/변경한다는 뜻.

---

## 왜 생성자를 private으로 막나?

생성자를 `public`으로 열면 외부에서 직접 `new`로 막 만들 수 있다:

```java
// 파라미터 순서 헷갈려도 컴파일은 됨 → 버그 유입 가능
new UserPrincipal(1L, "a@a.com", "name", "hash", "ROLE_ADMIN", List.of(), true);
new UserPrincipal(1L, "a@a.com", "name", null,   "ROLE_ADMIN", List.of(), true);
```

`private`으로 막고 팩토리 메서드만 열어두면:
- 객체를 만드는 방법이 통제된다
- 이름으로 **"어디서 왔는지"** 를 표현할 수 있다

---

## 팩토리 메서드란?

`private` 생성자를 내부에서 호출해서 객체를 반환하는 `public static` 메서드.
생성자를 직접 노출하지 않고 이름 있는 메서드로 감싼다.

```java
// "JWT 클레임으로부터 만든다"
public static UserPrincipal fromClaims(
        Long id, String email, String username,
        String roleCode, List<String> permissions
) {
    return new UserPrincipal(id, email, username, null, roleCode, permissions, true);
    //                                             ↑ passwordHash는 JWT에 없으니 null
}

// "DB 엔티티로부터 만든다"
public static UserPrincipal fromEntity(User u) {
    return new UserPrincipal(
        u.getId(), u.getEmail(), u.getUsername(),
        u.getPasswordHash(),  // ← DB에서 왔으니 실제 값 있음
        u.getRole().getCode(), List.of(), u.isActive()
    );
}
```

두 경우 내부 파라미터 구성이 다르다 (`fromClaims`는 passwordHash가 null, `fromEntity`는 실제 값).
생성자 하나로 열어두면 호출하는 쪽에서 어떤 값을 넣어야 하는지 알기 어렵다.

---

## UserPrincipal에서의 호출 흐름

```
필터(JwtAuthenticationFilter)
  → UserPrincipal.fromClaims(userId, email, username, roleCode, permissions) 호출
      → 내부에서 private 생성자 호출
      → UserPrincipal 객체 반환
  → SecurityContext에 등록
```

로그인 시 DB 인증:
```
CustomUserDetailsService
  → UserPrincipal.fromEntity(user) 호출
      → 내부에서 private 생성자 호출
      → UserPrincipal 객체 반환
```
