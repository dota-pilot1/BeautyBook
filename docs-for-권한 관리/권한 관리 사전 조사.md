## 2. 리프레시 토큰 — 추가 구현 필요한 것

### 현재 미구현 (리스크 있음)

| 항목 | 현황 | 필요 여부 |
|------|------|----------|
| **Refresh Token 재사용 감지** | 미구현 | ★★★ 필수 |
| **디바이스별 토큰 관리** | 미구현 | ☆ 선택 |
| **토큰 블랙리스트** | 미구현 | ☆ 선택 |

### 재사용 감지 (Refresh Token Reuse Detection)
현재 구조의 문제: 토큰이 탈취되어도 감지 불가능.

**구현 방법:**
```
rotate() 할 때 이미 사용된 토큰으로 재요청이 오면
→ 해당 유저의 모든 refresh_token 삭제 (강제 로그아웃)
→ 공격 시도로 간주
```

`RefreshToken` 테이블에 `previousToken` 컬럼 추가하거나,
`version` 필드로 단순하게 구현 가능.

**결론: 미용실 앱 수준에서는 현재 rotate 방식으로 충분. 나중에 추가.**

## 3. 권한 관리 전략 비교

### Option A: Role만 사용 (단순)

```java
public enum UserRole {
    ROLE_CUSTOMER,   // 일반 고객
    ROLE_STYLIST,    // 미용사
    ROLE_MANAGER,    // 매장 관리자
    ROLE_ADMIN       // 슈퍼 관리자
}
```

**장점:**
- 구현 간단, JWT 클레임에 바로 담김
- Spring Security `@PreAuthorize("hasRole('MANAGER')")` 바로 사용
- 미용실 앱 규모에서 충분

**단점:**
- 롤 추가 시 코드 변경 필요
- 세밀한 권한 제어 불가 (예: 특정 매장만 관리 가능)

**적합 상황:** MVP, 보일러플레이트 초기 버전

---

### Option B: Role + Permission 매핑 (중간)

```
Role ──── Permission (다대다)
ROLE_MANAGER → [BOOKING_READ, BOOKING_WRITE, STAFF_READ]
ROLE_STYLIST  → [BOOKING_READ, SCHEDULE_WRITE]
ROLE_CUSTOMER → [BOOKING_READ, BOOKING_WRITE_OWN]
```

**DB 구조:**
```sql
roles (id, name, description)
permissions (id, code, description)
role_permissions (role_id, permission_id)
user_roles (user_id, role_id)  -- 또는 User.role 단일 컬럼 유지
```

**장점:**
- 롤별 권한을 DB에서 관리 → UI로 변경 가능
- `@PreAuthorize("hasAuthority('BOOKING_WRITE')")` 사용 가능
- 보일러플레이트로서 범용성 높음

**단점:**
- JWT에 permission 목록 담으면 토큰 커짐 → 매 요청마다 DB 조회 or 캐시 필요
- 초기 구현 비용 높음

**적합 상황:** 관리자 UI에서 권한 조정이 필요한 경우

---

### Option C: RBAC 완전체 (복잡)

Resource + Action 기반:
```
BOOKING:READ, BOOKING:WRITE, STAFF:MANAGE, SALON:ADMIN
```
+ 테넌트(매장) 격리까지 포함.

**결론: 미용실 단일 매장 앱에서는 오버엔지니어링. 제외.**

## 4. 권장 구현 방향 (보일러플레이트 목표)

**Phase 1 — Role 기반 완성 (지금 할 것)**

```
UserRole: ROLE_CUSTOMER / ROLE_STYLIST / ROLE_MANAGER / ROLE_ADMIN
```

- 회원가입 시 기본 `ROLE_CUSTOMER` 부여
- 관리자 페이지에서 role 변경 API
- `@PreAuthorize` 적용 예시 API 포함
- 프론트: role별 메뉴/페이지 접근 제어

**Phase 2 — Permission 매핑 추가 (보일러플레이트 완성)**

```
Permission 테이블 + RolePermission 매핑 테이블
관리자 UI: 롤별 권한 토글 화면
```

- JWT에는 role만 담고, permission은 서버에서 캐시로 검증
- Redis or 인메모리 캐시 (Caffeine) 활용

## 5. 미용실 앱 롤 설계

```
ROLE_ADMIN
  ├── 전체 시스템 관리
  ├── 매장/직원 등록
  └── 모든 기능 접근

ROLE_MANAGER (매장 관리자)
  ├── 직원 스케줄 관리
  ├── 예약 전체 조회/변경
  └── 서비스(메뉴) 관리

ROLE_STYLIST (미용사)
  ├── 본인 스케줄 조회/변경
  ├── 본인 담당 예약 조회
  └── 예약 상태 변경 (IN_PROGRESS, COMPLETED)

ROLE_CUSTOMER (고객)
  ├── 예약 생성/취소 (본인 것만)
  ├── 서비스 목록 조회
  └── 예약 내역 조회 (본인 것만)
```
