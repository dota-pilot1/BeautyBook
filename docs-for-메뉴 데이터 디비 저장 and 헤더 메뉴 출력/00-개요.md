# 스텝 0: 개요 — 메뉴 DB화 + 헤더 출력 전체 플로우

> 기존 [Header.tsx](../../beauty-book--front/src/widgets/header/ui/Header.tsx) 의 네비게이션은 JSX 하드코딩이었다.
> 이 문서 세트는 **메뉴를 DB 테이블로 이관 → 공개 플랫 API → 프론트에서 트리로 재조립 → 재귀 렌더링 + 관리 UI** 로 전환하는 과정을 기록한다.
> 핵심 원칙: **저장·전송=플랫, 출력·관리 UI=트리**. 변환은 프론트 순수 함수 `buildTree(flat)` 하나로만.

> 이 문서들은 "계획" 이 아니라 **실제 구현이 반영된 결과 문서** 다. 설계 의도와 실제 구현이 다른 지점은 "결정 로그" 섹션에 남겼다.

---

## 왜 DB화 했는가

| 기존 | 문제 |
|------|------|
| 메뉴가 Header.tsx 에 JSX 로 하드코딩 | 메뉴 추가/수정 시 배포 필요 |
| 드롭다운 활성화 판별이 `pathname.startsWith(...)` 문자열 | 경로 변경 시 헤더 코드까지 고쳐야 함 |
| i18n 키만 번역 가능, 순서·노출 여부는 코드 수정 | 비개발자가 손댈 수 없음 |
| 1차/2차만 지원 | 3차 메뉴 필요 시 또 JSX 분기 |

---

## 실제 구현 범위

1. **N차 메뉴 (self-referencing)** — `parent_id` 자기참조로 무제한 depth.
2. **플랫 저장/전송** — DB·API 둘 다 플랫. `GET /api/menus` 한 번 호출로 전체 반환.
3. **프론트 트리 조립** — `buildTree(flat)` 로 재조립 → 재귀 렌더.
4. **관리 UI (트리 단독)** — DnD 로 형제 순서 변경, 클릭으로 상세 편집, 생성/삭제.
5. **권한 필터 (프론트)** — Header 의 `buildTree` 안에서 `requiredRole === userRole` 매칭.

---

## "플랫 ↔ 트리" 경계

```
┌────────────────────────────────────────────────────────────────┐
│  DB (menus 테이블)        ← 플랫 (id, parent_id, display_order) │
│      ↓ SELECT             저장 단위                              │
│  API GET /api/menus       ← 플랫 리스트 반환 (공개)              │
│      ↓ JSON               전송 단위                              │
│  프론트 useQuery          ← 플랫 받음                            │
│      ↓ filter(visible, requiredRole)   ← 필터는 여기서           │
│      ↓ buildTree(flat)                  ← 트리화                 │
│  <Header> / <MenuTreeTab> ← 트리 렌더                            │
└────────────────────────────────────────────────────────────────┘
```

규칙:
- **저장·전송 = 플랫**
- **관리 UI 편집 = 트리**
- **출력 = 트리 (재귀)**
- 플랫 → 트리 변환은 순수 함수 `buildTree(flat)`. Header 와 MenuTreeTab 에 각각 존재(간단해서 분리 안 함).
- 트리 → 플랫 변환 함수는 **없음**. 관리 UI 는 개별 row 단위 PATCH 로 저장.

---

## 실제 구현 순서 (진행된 대로)

1. **백엔드** (1-1 ~ 1-4)
2. **헤더 연동** (플랫 GET API + 클라이언트 buildTree)
3. **관리 UI — 테이블 버전** (MenuTable)
4. **관리 UI — 트리 버전 추가 (탭)** (MenuTreeTab, DnD + 상세 패널)
5. **리팩토링: 테이블 제거, 트리 단독** (최종)

### 회고 — 이 순서가 최적이었나

아니다. **관리 UI 부터 만들었어야 했다.**

- 시더로 데이터를 만든 덕에 헤더가 빨리 살아났지만, 시더 스키마가 굳은 뒤 관리 UI 를 붙이면서 **테이블 → 트리 → 테이블 제거** 로 중복 작업이 생겼다.
- 관리 UI 를 먼저 만들었으면 화면에서 바로 입력하며 필드/UX 요구를 확정한 뒤 헤더를 한 번에 맞출 수 있었다.
- 다만 시더가 없으면 "일단 관리 UI 로 진입할 권한 메뉴"가 없어 닭-달걀 문제가 생기니, **최소 시드 2~3개(대시보드 + 메뉴관리 자체) + 관리 UI 먼저** 가 이상적인 순서.

## 파일 구성 (결과)

### 백엔드

```
com.cj.beautybook.menu
├── domain/Menu.java                     @Entity self-referencing
├── infrastructure/MenuRepository.java   JpaRepository
├── application/MenuService.java         CRUD (트랜잭션 경계)
└── presentation/
    ├── MenuController.java              /api/menus (단일, 공개 GET + ADMIN 전용 CUD)
    └── dto/
        ├── MenuResponse.java
        ├── CreateMenuRequest.java
        └── UpdateMenuRequest.java

com.cj.beautybook.config/MenuSeeder.java  @Order(4) ApplicationRunner
```

### 프론트

```
src/entities/menu/
├── model/types.ts                   MenuRecord, MenuItem
└── api/menuApi.ts                   getAll/create/update/delete

src/features/menu-management/
├── MenuTreeTab.tsx                  트리 + DetailPanel + DnD (메인 UI)
└── MenuFormDialog.tsx               신규 생성 다이얼로그

src/app/menu-management/page.tsx     라우트 (RequireAuth + MenuTreeTab)
src/widgets/header/ui/Header.tsx     useQuery + buildTree (role 필터 포함)
```

## 설계 대비 실제 구현 차이 (요약)

| 주제 | 원래 계획 | 실제 구현 | 이유 |
|------|----------|-----------|------|
| 엔드포인트 분리 | `/api/menus` + `/api/admin/menus/**` 분리 | 단일 `/api/menus` + `@PreAuthorize` | 단순함 우선. 공개 GET 만 구별하면 충분 |
| 서버측 권한 필터 | `/api/menus` 가 유저별로 필터링된 결과 | 전체 반환, 프론트에서 필터 | 공개 캐싱 가능, 메뉴는 비밀 아님 |
| Reorder 전용 API | `PATCH /menus/reorder` | 개별 PATCH × N (Promise.all) | 메뉴 수 적어서 충분 |
| Parent 변경 전용 API | `PATCH /menus/{id}/parent` | 일반 `update()` 에 포함 | 동일 이유 |
| 변환 유틸 레이어 | `entities/menu/model/menuTree.ts` | Header / MenuTreeTab 인라인 | 코드 짧아 굳이 분리 안 함 |
| `flattenTree`, `pruneEmptyFolders`, `filterVisibleForUser` | 유틸 함수들 | **없음** | 실제 필요성 없었음 |
| 관리 UI | 테이블 + 다이얼로그 | 트리 + 우측 상세 패널 + 생성 다이얼로그 | DnD 로 순서 변경이 훨씬 직관적 |
| 순환 참조 방어 | 엔티티 `changeParent()` | **없음** | 현재는 UI 에서 부모를 루트만 선택 가능 → 순환 불가. 부모 자유 선택시 추가 필요 |
| 시더 메뉴 수 | 8개(권한관리 3차 포함) | 6개(권한관리 별도 페이지 아직 없음) | 현 화면 기준으로만 |

## 범위 밖 (v2 후보)

- 서버측 권한 필터 (유저별 `/api/menus` 필터)
- Reorder / changeParent 전용 엔드포인트 (메뉴 수 증가 시 필요)
- `@OneToMany children` + JOIN FETCH 최적화
- Redis 캐싱
- 키보드 DnD, 접근성
- 메뉴별 분석(클릭 카운트)
- 비로그인 사이트 프론트 메뉴, 사이드바 메뉴(`placement` 컬럼)
