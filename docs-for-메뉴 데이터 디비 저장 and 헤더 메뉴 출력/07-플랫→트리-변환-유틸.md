# 스텝 7: 플랫 → 트리 변환 — `buildTree`

> 서버는 플랫만 보낸다. 프론트는 **`buildTree(flat)` 로 트리화** 해서 렌더. 이 함수가 이 설계의 심장.
> **현재 구현은 별도 유틸 파일로 분리하지 않고 Header 와 MenuTreeTab 각각에 인라인** 으로 두고 있다.

---

## 배치 결정 (계획 대비 변경)

| 계획 | 실제 |
|------|------|
| `src/entities/menu/model/menuTree.ts` 에 `buildTree/filterVisibleForUser/flattenTree/pruneEmptyFolders` 유틸 | **없음**. `buildTree` 만 소비 지점 2곳에 인라인 |
| `MenuNode` 에 `depth` 필드 | **없음**. 재귀 렌더 파라미터로 depth 전달 |
| `filterVisibleForUser` 별도 함수 | **없음**. Header 의 `buildTree` 안에서 `visible && requiredRole === userRole` 인라인 필터 |
| `flattenTree` | **없음**. 관리 UI 가 개별 PATCH 를 쓰므로 필요 없음 |
| `pruneEmptyFolders` | **없음**. 현재 시더 기준 "빈 폴더" 케이스 없음 |

> 분리할 만큼 로직이 재사용되지 않아 유틸 레이어 생성을 생략. 둘로 갈라 두 개의 작은 함수로 유지.

## Header 의 `buildTree` ([Header.tsx](../../beauty-book--front/src/widgets/header/ui/Header.tsx))

**플랫 → 필터 → 트리** 가 한 함수에 있다.

```ts
function buildTree(flat: MenuRecord[], userRole: string | null): MenuItem[] {
  // 1. 필터 (visible + role)
  const visible = flat.filter(
    (m) => m.visible && (!m.requiredRole || m.requiredRole === userRole)
  );

  // 2. id → node map
  const map = new Map<number, MenuItem>();
  visible.forEach((m) => map.set(m.id, { ...m, children: [] }));

  // 3. parentId 로 연결
  const roots: MenuItem[] = [];
  map.forEach((item) => {
    if (item.parentId === null) roots.push(item);
    else map.get(item.parentId)?.children.push(item);
  });

  // 4. 형제 정렬
  const sort = (items: MenuItem[]) =>
    items.sort((a, b) => a.displayOrder - b.displayOrder);

  map.forEach((item) => sort(item.children));
  return sort(roots);
}
```

### 특징

- **필터·트리화 통합** — 순서: 필터 → 트리화. 권한 없는 부모의 자식은 부모가 없어 "고아" 가 되는데, 현재는 **`map.get(parentId)?.children.push(...)` 가 Optional chaining 이라 조용히 드롭**.
- **전이 규칙 (부모 제외 → 자식 제외) 자동**: 부모가 필터에서 빠지면 자식은 map 에 parent 없음 → children 에 추가되지 않고 roots 로도 안 감(parentId != null) → 결과적으로 드롭. 우연히 동작.
- **Role 비교**: 단순 `===`. 유저가 `ROLE_ADMIN` 이면 `ADMIN` 폴더와 자식이 보임. Permission 체크는 미구현.

## MenuTreeTab 의 `buildTree` ([MenuTreeTab.tsx](../../beauty-book--front/src/features/menu-management/MenuTreeTab.tsx))

**필터 없음, 전체 트리화** (관리 UI 는 숨김도 봐야 함).

```ts
function buildTree(flat: MenuRecord[]): MenuItem[] {
  const map = new Map<number, MenuItem>();
  flat.forEach((m) => map.set(m.id, { ...m, children: [] }));

  const roots: MenuItem[] = [];
  map.forEach((item) => {
    if (item.parentId === null) roots.push(item);
    else map.get(item.parentId!)?.children.push(item);
  });

  const sort = (items: MenuItem[]) => {
    items.sort((a, b) => a.displayOrder - b.displayOrder);
    items.forEach((i) => sort(i.children));   // 재귀 정렬
  };
  sort(roots);
  return roots;
}
```

Header 와 거의 동일. **차이: 필터 없음 + 재귀 정렬**(Header 는 Map 순회로 한 depth 만 정렬).

## 최종 파이프라인 (헤더)

```tsx
const { data: flatMenus = [] } = useQuery({ queryKey: ["menus"], queryFn: menuApi.getAll });
const userRole = user?.role?.code ?? null;
const tree = buildTree(flatMenus, userRole);

return tree.map(item => <NavItem key={item.id} item={item} />);
```

세 줄로 끝난다. "저장 = 플랫 → 전송 = 플랫 → 필터 = 플랫 → 트리화 → 렌더" 의 응축.

## 중복 제거는 언제?

현재 `buildTree` 가 두 곳에 존재하는 건 허용되는 중복 수준. 단, 다음 시점에 **`entities/menu/model/menuTree.ts`** 로 추출한다:

- 세 번째 소비 지점이 생길 때 (사이드바, 모바일 메뉴 등)
- `filterVisibleForUser` 에 permission 까지 포함시킬 필요가 생길 때
- 테스트가 필요해질 때 (단위 테스트하기엔 인라인이 불편)

YAGNI 원칙에 따라 지금은 그대로 둠.

## 범위 밖 (v2)

- `flattenTree(tree)` — 드래그로 재배열 후 서버 일괄 저장할 때 필요. 현재는 개별 PATCH 로 우회.
- `pruneEmptyFolders(tree)` — 권한 필터 후 자식이 전부 숨겨진 폴더 드롭다운을 제거. 현재는 해당 케이스 없음.
- `filterVisibleForUser(flat, user)` with permission set — permission 기반 필터 도입 시.
