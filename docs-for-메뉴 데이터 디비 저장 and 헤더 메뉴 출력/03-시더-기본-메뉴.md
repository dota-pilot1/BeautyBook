# 스텝 3: 시더 — 초기 메뉴 데이터

> 시더는 `code` 기준 **존재 시 건너뜀** (upsert-or-skip). 운영자가 관리 UI 에서 바꾼 값을 재기동 때 덮어쓰지 않는다.
> 실제 구현: [MenuSeeder.java](../../beauty-book-server/src/main/java/com/cj/beautybook/config/MenuSeeder.java)

---

## 실제 시드 데이터

| code | parent | label | labelKey | path | icon | requiredRole | order |
|------|--------|-------|----------|------|------|--------------|:-----:|
| `DASHBOARD` | — | 대시보드 | `nav.dashboard` | `/dashboard` | LayoutDashboard | — | 0 |
| `ADMIN` | — | 관리 | `nav.admin` | *(폴더형)* | Settings | `ROLE_ADMIN` | 1 |
| `ADMIN_USERS` | `ADMIN` | 유저 관리 | `nav.users` | `/users` | Users | `ROLE_ADMIN` | 0 |
| `ADMIN_ROLE_PERMISSIONS` | `ADMIN` | 역할-권한 매핑 | `nav.rolePermissions` | `/role-permissions` | ShieldCheck | `ROLE_ADMIN` | 1 |
| `ADMIN_SITE_SETTINGS` | `ADMIN` | 메인 관리 | `nav.siteSettings` | `/site-settings` | LayoutDashboard | `ROLE_ADMIN` | 2 |
| `ADMIN_MENU_MANAGEMENT` | `ADMIN` | 메뉴 관리 | `nav.menuManagement` | `/menu-management` | Menu | `ROLE_ADMIN` | 3 |

- `displayOrder` 는 **연속값(0,1,2,3,...)** 사용. (원래 계획인 10단위 띄우기는 미적용 — row 수가 적어 DnD 저장 시마다 재할당)
- 3차 메뉴는 **없음**. 실제 운용 구조와 일치시킴.
- `permission_categories`, `permissions` 화면이 아직 없어 해당 메뉴 시드도 **제외**. 추가 시 시더에 한 줄씩 넣으면 된다.

---

## 시더 구현 (실제)

```java
@Component
@Order(4)
@RequiredArgsConstructor
public class MenuSeeder implements ApplicationRunner {

    private final MenuRepository menuRepository;

    private record MenuDef(
            String code, String parentCode, String label, String labelKey,
            String path, String icon, String requiredRole, int displayOrder
    ) {}

    @Override
    @Transactional
    public void run(ApplicationArguments args) {
        List<MenuDef> defs = List.of(
            new MenuDef("DASHBOARD",             null,    "대시보드",       "nav.dashboard",       "/dashboard",        "LayoutDashboard", null,                    0),
            new MenuDef("ADMIN",                 null,    "관리",           "nav.admin",           null,                "Settings",        RoleSeeder.ROLE_ADMIN,   1),
            new MenuDef("ADMIN_USERS",           "ADMIN", "유저 관리",      "nav.users",           "/users",            "Users",           RoleSeeder.ROLE_ADMIN,   0),
            new MenuDef("ADMIN_ROLE_PERMISSIONS","ADMIN", "역할-권한 매핑", "nav.rolePermissions", "/role-permissions", "ShieldCheck",     RoleSeeder.ROLE_ADMIN,   1),
            new MenuDef("ADMIN_SITE_SETTINGS",   "ADMIN", "메인 관리",      "nav.siteSettings",    "/site-settings",    "LayoutDashboard", RoleSeeder.ROLE_ADMIN,   2),
            new MenuDef("ADMIN_MENU_MANAGEMENT", "ADMIN", "메뉴 관리",      "nav.menuManagement",  "/menu-management",  "Menu",            RoleSeeder.ROLE_ADMIN,   3)
        );

        for (MenuDef def : defs) {
            if (menuRepository.existsByCode(def.code())) continue;
            Menu parent = def.parentCode() != null
                    ? menuRepository.findByCode(def.parentCode()).orElse(null)
                    : null;
            menuRepository.save(Menu.create(
                    def.code(), parent, def.label(), def.labelKey(),
                    def.path(), def.icon(), false,
                    def.requiredRole(), null, true, def.displayOrder()
            ));
        }
    }
}
```

---

## 중요 원칙

1. **`existsByCode` 스킵** — 이미 있으면 건드리지 않음. 운영자 수정본 보존.
2. **부모 먼저, 자식 나중** — `List.of(...)` 선언 순서가 곧 실행 순서. 루트(`DASHBOARD`, `ADMIN`) 먼저.
3. **`@Order(4)` — Role/Permission 시더 뒤** — `requiredRole="ROLE_ADMIN"` 값이 의미를 가지려면 `RoleSeeder` (@Order 더 낮음)가 먼저. FK 는 아니지만 의미적 의존.
4. **시더는 삭제 안 함** — 시드에서 빠진 `code` 가 DB 에 있어도 그대로 둔다. 삭제는 관리 UI 로만.

---

## 초기 구현 시 있었던 버그 (기록)

- **증상**: 시더가 `DASHBOARD` 만 체크하고 전체를 early-return → 새 메뉴가 추가되지 않음.
- **원인**: 초기 구현이 `if (existsByCode("DASHBOARD")) return;` 식으로 전체 스킵.
- **수정**: `MenuDef` record 순회하며 **개별 `existsByCode` 체크** 로 바꿈. 시더에 한 줄 추가하면 새 code 만 추가되고 나머지는 건드리지 않음.

---

## 메뉴 추가 시 절차

1. `MenuSeeder.run()` 의 `defs` 리스트에 `new MenuDef(...)` 한 줄 추가.
2. 서버 재기동 → 새 `code` 만 insert.
3. 운영 DB 에서 순서/라벨 바꾸고 싶으면 **시더 대신 관리 UI** 에서 수정.
4. code 리네이밍은 별도 마이그레이션 SQL (시더에서 바꾸면 기존 row 와 충돌).
