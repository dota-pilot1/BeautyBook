## Phase 1 — 필수 보완 (런칭 전 블로커, 약 2주)

### 1. Refresh Token → httpOnly Cookie 전환

**현재:** `localStorage`에 refresh token 저장  
**문제:** XSS 공격으로 토큰 탈취 가능 → 세션 하이재킹 위험  
**해결:**
- 백엔드: `/api/auth/refresh` 응답 시 `Set-Cookie: refreshToken=...; HttpOnly; Secure; SameSite=Strict`
- 프론트엔드: `tokenStorage.ts`에서 refresh token 관련 localStorage 로직 제거, 쿠키는 브라우저가 자동 첨부
- Spring Security CORS 설정에 `allowCredentials(true)` 확인

---

### 2. 이메일 인증 플로우

**현재:** 회원가입 즉시 `active = true`로 활성화  
**문제:** 존재하지 않는 이메일로 계정 생성 가능, 스팸 가입 무방비  
**해결:**
- `User` 엔티티에 `emailVerified: boolean`, `verificationToken: String` 필드 추가
- 가입 시 이메일 발송 → 토큰 클릭 → `active = true` 전환
- Spring Mail (SMTP) 또는 AWS SES 연동
- 미인증 계정은 로그인 거부 처리

---

### 3. Rate Limiting (API 레벨)

**현재:** API 호출 횟수 제한 없음  
**문제:** 로그인 브루트포스, 회원가입 스팸, DDoS 취약  
**해결:**
- `spring-boot-starter-data-redis` + Bucket4j 라이브러리 조합
- `/api/auth/login`, `/api/auth/signup`에 IP별 분당 호출 제한 적용
- 또는 AWS API Gateway / Nginx `limit_req_zone` 인프라 레벨 처리

---

### 4. `ddl-auto: update` → `validate` 전환 + Flyway 도입

**현재:** `spring.jpa.hibernate.ddl-auto: update`  
**문제:** 운영 서버에서 Hibernate가 DDL을 자동 변경 → 데이터 손실 위험  
**해결:**
```yaml
# application-prod.yaml
spring:
  jpa:
    hibernate:
      ddl-auto: validate
  flyway:
    enabled: true
    locations: classpath:db/migration
```
- `src/main/resources/db/migration/V1__init_schema.sql` 작성 (현재 스키마 기준)
- 이후 변경 시 `V2__add_email_verified.sql` 형태로 추가

---

### 5. 환경변수 분리 (dev / prod 프로파일)

**현재:** `application.yaml` 단일 파일에 dev 설정 혼재  
**문제:** JWT 시크릿, DB 비밀번호가 코드에 하드코딩될 위험  
**해결:**
```
application.yaml          # 공통
application-dev.yaml      # 로컬 개발 (postgres/postgres, 약한 JWT 시크릿)
application-prod.yaml     # 운영 (환경변수로 주입: ${DB_PASSWORD}, ${JWT_SECRET})
```
- 프론트엔드: `.env.local` (dev), `.env.production` (prod) 분리

## Phase 2 — 권장 보완 (품질 강화, 약 3주)

### 6. 테스트 코드 작성

**현재:** `BeautyBookServerApplicationTests` 컨텍스트 로드 테스트만 존재

**우선순위 높은 테스트:**

```
백엔드 (JUnit 5 + Spring Boot Test)
  - AuthServiceTest: 회원가입, 로그인, 토큰 갱신, 로그아웃
  - RoleServiceTest: CRUD, 시스템 역할 보호 로직
  - JwtTokenProviderTest: 발급, 파싱, 만료 처리
  - GlobalExceptionHandlerTest: 각 ErrorCode 응답 형식

프론트엔드 (Playwright E2E)
  - 로그인 성공 / 실패
  - 권한 없는 페이지 접근 차단 (RequireAuth 가드)
  - 토큰 만료 시 자동 갱신 흐름
```

---

### 7. GitHub Actions CI/CD

**현재:** 수동 빌드/배포  
**해결:**
```yaml
# .github/workflows/ci.yml
on: [push, pull_request]
jobs:
  backend-test:
    - ./gradlew test
  frontend-lint-build:
    - npm run lint && npm run build
```
- PR 머지 전 테스트 통과 필수화
- `main` 브랜치 푸시 시 Docker 이미지 빌드 + 레지스트리 푸시 자동화

---

### 8. 프론트엔드 Dockerfile + Docker Compose 통합

**현재:** 백엔드/DB만 Docker Compose, 프론트엔드는 수동 실행  
**해결:**
```dockerfile
# beauty-book--front/Dockerfile
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

FROM node:20-alpine AS runner
WORKDIR /app
COPY --from=builder /app/.next/standalone ./
EXPOSE 4100
CMD ["node", "server.js"]
```
- `docker-compose.yml`에 `frontend` 서비스 추가 → `docker compose up` 한 번으로 전체 실행

---

### 9. 감사 로그 (Audit Log)

**현재:** 변경 이력 추적 불가  
**문제:** 엔터프라이즈 고객 요구사항 (누가 언제 무엇을 바꿨는가)  
**해결:**
- Spring Data JPA `AuditorAware` + `@EnableJpaAuditing`
- `createdBy`, `updatedBy` 필드를 공통 `BaseEntity`에 추가
- 또는 별도 `AuditLog` 테이블 + AOP로 서비스 레이어 변경 감지

## Phase 3 — 선택 보완 (필요 시)

| 항목 | 필요한 경우 |
|------|------------|
| **OAuth2** (Google / Kakao / Naver) | B2C 서비스, 소셜 로그인 필요 시 |
| **Redis 캐시** | 권한 조회 빈번 시 (역할/권한 캐싱), 세션 공유 |
| **다중 테넌시 (Multi-tenancy)** | 솔루션을 여러 고객사에 SaaS 형태로 납품 시 |
| **Kafka / 이벤트 버스** | 마이크로서비스 전환, 이벤트 드리븐 아키텍처 |
| **2FA (TOTP)** | 금융/의료 등 고보안 요구 시 |
| **OpenTelemetry + Grafana** | 운영 모니터링, 분산 추적 |
