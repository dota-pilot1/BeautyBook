# 공통 에러 처리 구조 분석

## 파일 구조

```
common/
├── exception/
│   ├── ErrorCode.java                  # 에러코드 enum (HTTP 상태 + 코드 + 메시지)
│   ├── BusinessException.java          # 도메인 예외 베이스 클래스
│   ├── DuplicateEmailException.java    # 구체 도메인 예외
│   ├── InvalidRefreshTokenException.java
│   └── GlobalExceptionHandler.java     # @RestControllerAdvice (예외 → HTTP 응답 변환)
└── response/
    └── ErrorResponse.java              # API 에러 응답 DTO
```

---

## DDD 관점에서의 구조 평가

### 잘 된 부분

**1. 도메인 예외를 의미 있는 이름으로 분리**

```java
// 예외를 던지는 곳 (도메인/애플리케이션 레이어)
throw new DuplicateEmailException();
throw new InvalidRefreshTokenException();
```

`throw new BusinessException(ErrorCode.DUPLICATE_EMAIL)` 대신 의미 있는 클래스를 별도로 두어,
던지는 코드만 봐도 어떤 도메인 규칙이 위반됐는지 바로 알 수 있음.

**2. BusinessException이 도메인 예외의 단일 진입점**

```java
public class BusinessException extends RuntimeException {
    private final ErrorCode errorCode;
}
```

모든 도메인 예외가 `BusinessException`을 상속 → `GlobalExceptionHandler`에서 단 하나의 핸들러로 처리 가능.
새 도메인 예외 추가 시 `BusinessException`만 상속하면 자동으로 처리 체계에 편입됨.

**3. 예외 변환 책임이 Presentation 레이어에만 존재**

```
도메인/애플리케이션 레이어  →  예외 throw (HTTP 모름)
       ↓
GlobalExceptionHandler  →  예외를 HTTP 응답으로 변환 (Presentation 관심사)
```

도메인 로직이 HTTP를 직접 다루지 않음.

**4. 에러 응답 형식 통일**

```json
{
  "code": "AUTH_001",
  "message": "이미 사용 중인 이메일입니다.",
  "timestamp": "2026-04-21T10:00:00Z",
  "fieldErrors": null
}
```

모든 에러가 동일한 형식으로 반환되어 프론트에서 일관된 처리 가능.

---

### 아쉬운 부분 (실무에서 논의 포인트)

**1. ErrorCode에 HttpStatus가 포함됨**

```java
DUPLICATE_EMAIL(HttpStatus.CONFLICT, "AUTH_001", "이미 사용 중인 이메일입니다.")
```

순수 DDD 관점에서는 도메인 레이어가 HTTP 개념(`HttpStatus`)을 알면 안 됨.
그러나 실용적으로는 대부분의 프로젝트에서 이 패턴을 씀 — 허용 가능한 타협.

순수하게 분리하려면:
```
ErrorCode (HTTP 모름)  →  ExceptionHandler에서 ErrorCode → HttpStatus 매핑
```
하지만 코드가 늘어나고 관리 포인트가 증가해 실익이 적음.

**2. 도메인 예외가 현재 모두 `common`에 있음**

`DuplicateEmailException`, `InvalidRefreshTokenException`은 auth/user 도메인의 예외인데 `common`에 위치.
도메인이 커지면 `user/exception/`, `voice/exception/` 으로 분리하는 것이 더 DDD스러움.

현재는 도메인이 작으므로 `common` 통합 관리가 실용적. **도메인이 3개 이상으로 늘어나는 시점에 분리 권장.**

## 에러 추가 방법 (패턴)

새로운 도메인 예외를 추가할 때의 절차:

### 1. ErrorCode에 코드 추가

```java
// ErrorCode.java
CALL_NOT_FOUND(HttpStatus.NOT_FOUND, "CALL_001", "상담 기록을 찾을 수 없습니다."),
```

### 2. 구체 예외 클래스 생성

```java
public class CallNotFoundException extends BusinessException {
    public CallNotFoundException() {
        super(ErrorCode.CALL_NOT_FOUND);
    }
}
```

### 3. 도메인/서비스에서 throw

```java
// VoiceService.java
callRepository.findByCallSid(callSid)
    .orElseThrow(CallNotFoundException::new);
```

`GlobalExceptionHandler`는 수정 불필요 — `BusinessException`을 자동으로 잡아서 처리.

---

## 전체 흐름 요약

```
클라이언트 요청
    ↓
Controller (Presentation)
    ↓
Service/Domain (도메인 규칙 위반 시 예외 throw)
    ↓  예: throw new DuplicateEmailException()
GlobalExceptionHandler (@RestControllerAdvice)
    ↓  BusinessException → ErrorResponse 변환
클라이언트 응답 { code, message, timestamp, fieldErrors }
```

## 결론

현재 구조는 **실용적인 DDD 스타일로 잘 구성**되어 있음.

| 항목 | 평가 |
|------|------|
| 도메인 예외 분리 | ✅ BusinessException 계층 구조 |
| HTTP 관심사 분리 | ✅ GlobalExceptionHandler에만 집중 |
| 응답 형식 통일 | ✅ ErrorResponse DTO |
| 확장성 | ✅ 새 예외 추가 시 핸들러 수정 불필요 |
| 순수 DDD 준수 | ⚠️ ErrorCode에 HttpStatus 포함 (실용적 타협) |
| 도메인별 패키지 분리 | ⚠️ 현재 common 통합 (도메인 증가 시 분리 권장) |
