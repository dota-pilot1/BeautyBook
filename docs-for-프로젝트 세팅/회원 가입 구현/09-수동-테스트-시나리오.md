# 9. 수동 테스트 시나리오

## 준비
- Postgres 기동: `docker compose up -d`
- 백엔드 기동: `./gradlew bootRun` (또는 IntelliJ Run)
- 프론트 기동: `cd twilio-callcenter-front && npm run dev`
- 브라우저: `http://localhost:4100/register`

## 백엔드 API 단독 테스트 (curl)

### ✅ 정상 가입 → 201
```bash
curl -i -X POST http://localhost:4101/api/auth/signup \
  -H 'Content-Type: application/json' \
  -d '{"email":"alice@example.com","password":"Alice123!@","username":"앨리스"}'
```
기대:
- `HTTP/1.1 201 Created`
- body: `{"id":1,"email":"alice@example.com","username":"앨리스","role":"ROLE_USER","createdAt":"..."}`

### ❌ 중복 이메일 → 409
같은 명령 재실행. 기대:
- `HTTP/1.1 409 Conflict`
- body: `{"code":"AUTH_001","message":"이미 사용 중인 이메일입니다.","timestamp":"...","fieldErrors":null}`

### ❌ 검증 실패 (짧은 비번) → 400
```bash
curl -i -X POST http://localhost:4101/api/auth/signup \
  -H 'Content-Type: application/json' \
  -d '{"email":"bob@example.com","password":"short","username":"밥"}'
```
기대:
- `HTTP/1.1 400 Bad Request`
- body `fieldErrors.password`: "비밀번호는 영문/숫자/특수문자를 포함한 8자 이상이어야 합니다."

### ❌ 검증 실패 (이메일 형식) → 400
```bash
curl -i -X POST http://localhost:4101/api/auth/signup \
  -H 'Content-Type: application/json' \
  -d '{"email":"not-an-email","password":"Good123!@","username":"홍"}'
```
기대:
- `HTTP/1.1 400 Bad Request`
- body `fieldErrors.email`: "올바른 이메일 형식이 아닙니다."

### ❌ 검증 실패 (사용자명 1자) → 400
```bash
curl -i -X POST http://localhost:4101/api/auth/signup \
  -H 'Content-Type: application/json' \
  -d '{"email":"c@c.com","password":"Good123!@","username":"짧"}'
```
기대: `fieldErrors.username`: "사용자명은 2~50자여야 합니다."

### ✅ 이메일 중복 체크
```bash
# 사용 가능
curl -i 'http://localhost:4101/api/auth/check-email?email=new@example.com'
# → {"available":true}

# 이미 사용 중
curl -i 'http://localhost:4101/api/auth/check-email?email=alice@example.com'
# → {"available":false}

# 잘못된 형식
curl -i 'http://localhost:4101/api/auth/check-email?email=invalid'
# → 400 fieldErrors 없이 ConstraintViolationException 메시지
```

## DB 검증
```bash
docker exec -it twilio-callcenter-postgres psql -U postgres -d twilio_callcenter \
  -c "SELECT id, email, username, role, active, created_at FROM users;"
```
- `password_hash` 가 `$2a$10$...` BCrypt 포맷인지:
  ```sql
  SELECT password_hash FROM users WHERE email='alice@example.com';
  ```
- BCrypt 원문 비밀번호가 평문 저장되지 않았는지 반드시 확인

## 프론트 E2E (수동)

| # | 입력 | 기대 |
| --- | --- | --- |
| 1 | 빈 제출 | 모든 필드에 zod 에러 메시지 노출 |
| 2 | 이메일: `abc`, 나머지 정상 | email 필드: "올바른 이메일 형식이 아닙니다." |
| 3 | 비번 `short`, 확인도 동일 | password 필드: 복잡도 경고 |
| 4 | 비번/확인 불일치 | passwordConfirm 필드: "비밀번호가 일치하지 않습니다." |
| 5 | 모두 정상, 새 이메일 | sonner toast "회원가입이 완료되었습니다." + `/login` 리다이렉트 |
| 6 | 이미 쓴 이메일 재시도 | email 필드에 "이미 사용 중인 이메일입니다." (서버 `AUTH_001` 매핑) |
| 7 | 백엔드 내림 상태 제출 | toast error "요청 중 오류가 발생했습니다." |

## Pass 기준
- 위 API 테스트 **전부 통과**
- DB 에 BCrypt 해시 저장 확인
- 프론트 E2E 시나리오 7개 전부 기대대로 동작
- `/api/auth/signup`, `/api/auth/check-email` 외의 경로는 401 반환 (시큐리티 체인이 막고 있는지)

## 완료 후
"로그인 구현" 문서로 이동. 이 시점의 `users` 테이블 데이터를 지우지 말 것 — 로그인 테스트에 그대로 사용.
