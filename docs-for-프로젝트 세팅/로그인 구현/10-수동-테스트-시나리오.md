# 10. 수동 테스트 시나리오

## 준비
- 회원가입 단계에서 만든 테스트 계정 최소 1개 존재 (`alice@example.com` / `Alice123!@`)
- Postgres, 백엔드, 프론트 전부 기동

## 백엔드 API 단독 (curl)

### 1. 정상 로그인 → 200 + 토큰 2종
```bash
curl -s -X POST http://localhost:4101/api/auth/login \
  -H 'Content-Type: application/json' \
  -d '{"email":"alice@example.com","password":"Alice123!@"}' | tee /tmp/login.json | jq .
ACCESS=$(jq -r .accessToken /tmp/login.json)
REFRESH=$(jq -r .refreshToken /tmp/login.json)
```
기대:
- `accessToken`, `refreshToken` 각각 `eyJ...` 형식
- `user.role`, `user.email` 등 포함
- `accessTokenExpiresInSec: 1800`

### 2. 토큰으로 `/me` 호출 → 200
```bash
curl -i http://localhost:4101/api/auth/me -H "Authorization: Bearer $ACCESS"
```

### 3. 잘못된 비번 → 401 AUTH_003
```bash
curl -i -X POST http://localhost:4101/api/auth/login \
  -H 'Content-Type: application/json' \
  -d '{"email":"alice@example.com","password":"wrong"}'
```
- body: `{"code":"AUTH_003","message":"이메일 또는 비밀번호가 올바르지 않습니다.",...}`

### 4. 존재하지 않는 이메일 → **동일한 401 AUTH_003**
```bash
curl -i -X POST http://localhost:4101/api/auth/login \
  -H 'Content-Type: application/json' \
  -d '{"email":"nobody@example.com","password":"Alice123!@"}'
```
> **정보 누출 방지**: 비번 오류와 이메일 오류가 같은 응답이어야 함 (브루트포스 시 유저 존재 여부 알려주지 않음)

### 5. 토큰 없이 `/me` → 401
```bash
curl -i http://localhost:4101/api/auth/me
```

### 6. 위조 토큰 → 401
```bash
curl -i http://localhost:4101/api/auth/me -H "Authorization: Bearer fake.token.here"
```

### 7. Refresh 토큰으로 `/me` 접근 → 401 (필터가 ACCESS 만 통과)
```bash
curl -i http://localhost:4101/api/auth/me -H "Authorization: Bearer $REFRESH"
```

### 8. 정상 refresh → 새 토큰 발급 + 기존 refresh 무효화
```bash
curl -s -X POST http://localhost:4101/api/auth/refresh \
  -H 'Content-Type: application/json' \
  -d "{\"refreshToken\":\"$REFRESH\"}" | tee /tmp/refresh.json | jq .
NEW_ACCESS=$(jq -r .accessToken /tmp/refresh.json)
NEW_REFRESH=$(jq -r .refreshToken /tmp/refresh.json)
```
- `NEW_ACCESS != ACCESS`, `NEW_REFRESH != REFRESH` 이어야 함

### 9. 이전 refresh 재사용 → 401 AUTH_006 (회전 감지)
```bash
curl -i -X POST http://localhost:4101/api/auth/refresh \
  -H 'Content-Type: application/json' \
  -d "{\"refreshToken\":\"$REFRESH\"}"
```
- AUTH_006

### 10. 로그아웃 → 204
```bash
curl -i -X POST http://localhost:4101/api/auth/logout -H "Authorization: Bearer $NEW_ACCESS"
```

### 11. 로그아웃 후 refresh 시도 → 401 (DB에서 삭제됨)
```bash
curl -i -X POST http://localhost:4101/api/auth/refresh \
  -H 'Content-Type: application/json' \
  -d "{\"refreshToken\":\"$NEW_REFRESH\"}"
```

### 12. DB 확인
```bash
docker exec -it twilio-callcenter-postgres psql -U postgres -d twilio_callcenter \
  -c "SELECT user_id, length(token), expires_at FROM refresh_tokens;"
```
- 로그아웃 후엔 해당 user_id row 가 없어야 함

## 프론트 E2E (브라우저)

| # | 시나리오 | 기대 |
| --- | --- | --- |
| 1 | `/dashboard` 직접 접속 (비로그인) | `/login?next=%2Fdashboard` 로 리다이렉트 |
| 2 | 로그인 성공 | sonner toast + `/dashboard` 이동, 유저명 표시 |
| 3 | 새로고침 | 토큰 복원 → 로그인 유지 (네트워크 탭에 `/api/auth/me` 호출) |
| 4 | localStorage 에서 `twilio.accessToken` 만 삭제 후 새로고침 | refresh 성공 → 로그인 유지 |
| 5 | 둘 다 삭제 후 새로고침 | `/login` 이동 |
| 6 | 잘못된 비번 | password 필드 아래에 "이메일 또는 비밀번호가 올바르지 않습니다." |
| 7 | 대시보드에서 로그아웃 버튼 | toast 없이 `anonymous` → `/login` |
| 8 | 로그인 → 백엔드에서 강제로 refresh_tokens row 삭제 → 30분 후(또는 access 토큰 만료 시뮬) | refresh 실패 → `/login` 자동 이동 |
| 9 | 로그인 → DevTools Application 탭에서 `twilio.accessToken` 을 `abc.def.ghi` 쓰레기로 조작 → 아무 보호 API 호출 | 인터셉터가 refresh 시도 → 성공 시 재시도 OK, 실패 시 /login |
| 10 | `/login?next=/evil.example.com` | 로그인 성공해도 외부로 안 나감 — `/dashboard` 로 가야 함 (오픈 리다이렉트 방지) |

## Pass 기준
- 백엔드 12개 시나리오 전부 기대대로
- 프론트 10개 시나리오 전부 기대대로
- 토큰 회전 동작 확인 (이전 refresh 재사용 시 거부)
- CORS 문제 없이 브라우저에서 `Authorization` 헤더가 실제 요청에 붙어있는지 DevTools 네트워크 탭에서 확인

## 알려진 한계 (나중 과제)
1. **httpOnly 쿠키로 refresh 이전** — XSS 노출 감소
2. **Rate limiting** — /login 브루트포스 방어 (Resilience4j 또는 Bucket4j)
3. **Account lockout** — 로그인 N회 연속 실패 시 잠금
4. **세션 목록/디바이스 관리** — 현재는 1 user = 1 refresh row (멀티 디바이스 로그인 시 기존 디바이스가 튕김)
5. **SSR 보호** — middleware.ts 로 서버 사이드 가드 추가 시 정적 export 와의 호환성 검토
6. **이메일 인증 / 비번 재설정 / OAuth2** — 별도 스프린트

## 완료 후
이 시점에서 회원가입 + 로그인 + 토큰 리프레시 + 로그아웃 전체 플로우가 동작. 이후 기능 개발은 모두 `RequireAuth` 로 감싸고 `api` 인스턴스를 쓰면 자동 인증.
